# Ignition 8.3 Seedability Matrix

This matrix records which Ignition 8.3 connection types you can provision from the filesystem and environment, versus which still require manual entry in the gateway UI. Every downstream phase of `ignition-stack` reads this contract to decide what its wizard can promise and what it has to defer to `POST-SETUP.md`.

The matrix has two verdict columns because the *config* and *secret* parts of a connection often split: the config (URL, port, username) is usually file-seedable, but the secret (password, API key) may need a separate mechanism. Recording them separately lets the service catalog seed the config part while the post-setup generator covers the secret part.

> **Status:** investigation complete for Phase 1. Verdicts here reflect what was observed on a live `inductiveautomation/ignition:8.3.6` gateway booted from `scripts/seeding-poc/baseline/` with the `inductive-automation/template-ignition-project` resources copied in. The 17 screenshots under `scripts/seeding-poc/screenshots/baseline/` and the module-install evidence under `scripts/seeding-poc/screenshots/module-install/` back the rows below.

## Investigation target

- Image: `inductiveautomation/ignition:8.3.6`
- Resources tree: `data/config/resources/{external,core,local,<mode>}` with `<mode>` chosen by `-Dignition.config.mode=<mode>` JVM arg. The reference template uses `dev` (which inherits from `core`, which inherits from `external`, per each tree's `config-mode.json`).
- Path template observed: `data/config/resources/<mode>/<plugin-id>/<subsystem>/<item-name>/{config.json,resource.json}` where `<plugin-id>` is `ignition` for platform subsystems and `com.inductiveautomation.<module>` for module-owned subsystems (the Cirrus Link modules use `com.cirruslink.<module>` once installed).
- `resource.json` carries `attributes.uuid`, `attributes.lastModification.{actor,timestamp}`, and `attributes.lastModificationSignature` (a 64-char hex blob). Inspection of the template shows the gateway accepts these as written; whether the signature is validated strictly is **not** load-bearing in practice because a JWE-encrypted secret from the template's `internal-secret-provider` config decoded cleanly on a fresh 8.3.6 gateway (see `screenshots/baseline/db-connection.png`, status VALID). That outcome rules out per-gateway encryption keys for the internal secret provider, which means the matrix's "secret" column collapses to a single answer for any subsystem that references a secret through `{"providerName": "internal-secret-provider", ...}`: the secret value is file-seedable as a JWE blob lifted from another file-seeded gateway.

## Module install verdict (q-module-install)

**Result:** all three options - (a) runtime volume-mount, (b) derived `Dockerfile` `COPY`, (c) bootstrap-container copy - place the `.modl` correctly. None of them is *sufficient* on its own.

**Env var:** `GATEWAY_MODULES_ENABLED` is the gateway-loading variable. Format: comma-delimited *fully-qualified module identifiers*, NOT paths. For the test `.modl` (`MQTT-Engine-signed-5.0.2.modl`) the value is `com.cirruslink.mqtt.engine.gateway`, taken verbatim from the module's `module.xml`. Filename and path forms tested separately both fail to enable the module. The docs page for platform environment variables describes this var as "a comma-delimited set of built-in and third party fully-qualified module identifiers" - that phrasing is correct; the venture context's "fully-qualified in-container path" expectation is wrong and needs to be updated everywhere downstream.

**The new constraint:** with `GATEWAY_MODULES_ENABLED` set and the `.modl` placed under `/usr/local/bin/ignition/user-lib/modules/`, the gateway boots into state `RUNNING` with details `COMMISSIONING` and the log line `Resources needing commissioning: modules`. The commissioning UI at `/welcome` shows the third-party module as `NOT TRUSTED` (certificate) and `NOT ACCEPTED` (license). These two acceptance steps gate activation. No environment variable observed in the 8.3.6 docker image's `docker-entrypoint.sh` or the platform env-var docs bypasses them. Evidence in `scripts/seeding-poc/screenshots/module-install/c1-wizard.png`.

**What this means for downstream phases:** placing a third-party `.modl` from the filesystem alone leaves the gateway pending UI commissioning. Phase 2+ must either (i) automate the commissioning wizard (Playwright-style scripted acceptance during `init`), (ii) defer third-party module trust and license acceptance to `POST-SETUP.md`, or (iii) investigate pre-populating the gateway's certificate trust store and module license records from the filesystem - a follow-up that did not fit in Phase 1.

## Matrix

Verdict legend: `yes` = file-seeding works without UI touch; `partial` = some aspects work, some don't (the fallback column says where the partial-ness lives); `no` = manual UI entry required.

| connection-type | file-seedable-config | file-seedable-secret | path-template / manual fallback |
| --- | --- | --- | --- |
| `db-connection` | yes | yes | Config: `resources/<mode>/ignition/database-connection/<name>/{config.json,resource.json}`. Secret: referenced from config via `{"providerName":"internal-secret-provider","secretName":"..."}` and stored as a JWE blob in `resources/<mode>/ignition/secret-provider/internal-secret-provider/config.json` under `settings.secrets.<name>.ciphertext`. The template's JWE values decoded cleanly on a fresh gateway, so this is fully file-seedable. Evidence: `screenshots/baseline/db-connection.png` (status VALID). |
| `jdbc-driver` | yes | n/a | Driver metadata: `resources/<mode>/ignition/database-driver/<Name>/config.json` (classname, URL format, default translator, default validation query). The `.jar` itself must be placed in `user-lib/jdbc/` either via bootstrap copy or derived Dockerfile - the template ships configs for PostgreSQL, MySQL, MariaDB, MSSQL, Oracle, SQLite. Evidence: `screenshots/baseline/jdbc-driver.png`. |
| `database-translator` | yes | n/a | `resources/<mode>/ignition/database-translator/<TYPE>/config.json`. Built-ins (POSTGRES, MYSQL, MSSQL, ORACLE, SQLITE, GENERIC) ship with the gateway; the template overlays an ORACLE entry, confirming custom translators are file-seedable the same way. | `opc-ua-connection` | yes | yes (via secret-provider) | Outbound OPC-UA client connections to external servers. Path: `resources/<mode>/com.inductiveautomation.opcua/connection/<name>/{config.json,resource.json}` (the connection plugin namespace). Secrets (usernames/passwords for the remote server) follow the same `internal-secret-provider` indirection as `db-connection`. | `opc-ua server-config` | yes | n/a | `resources/<mode>/com.inductiveautomation.opcua/{server-config,access-control,one-time}/config.json`. Anonymous-access, bind ports, security policies, and access-control rules are all file-seedable. Evidence: `screenshots/baseline/opc-ua-server-config.png`. |

Adding an OIDC or SAML provider means a new directory at the same path with `profile.type=oidc` or `saml` and the type-specific config fields; client secrets reference the internal-secret-provider the same way `db-connection` does. | `tag-provider` | yes | n/a | `resources/<mode>/ignition/tag-provider/<name>/{config.json,resource.json}`. Phase 1 confirmed the project tree is read on boot; per-resource type behavior was not deeply exercised but matches the validated template's pattern. |
| `historian-provider` | yes | n/a | `resources/<mode>/com.inductiveautomation.historian/historian-provider/<name>/{config.json,resource.json}`. The template ships a `db`-backed historian provider that reads as configured on a fresh gateway. Evidence: `screenshots/baseline/historian-provider.png`. |
| `jvm-arg` | yes | n/a | JVM args go on the gateway container `command:` line after the standalone `--` separator. Example from the template: `-Dignition.config.mode=${DEPLOYMENT_MODE:-dev}`. No env-var wrapper is needed; the compose `command:` is the seam. |
| `env-var` | yes | n/a | Standard container env. Confirmed: `ACCEPT_IGNITION_EULA=Y`, `GATEWAY_ADMIN_USERNAME` (defaults to `admin`), `GATEWAY_ADMIN_PASSWORD`, `IGNITION_EDITION=standard|edge|maker`, `IGNITION_LICENSE_KEY`, `IGNITION_ACTIVATION_TOKEN`, `IGNITION_ROOT_KEY_PASSWORD_FILE`, `GATEWAY_MODULES_ENABLED`. The `<VAR>_FILE` secret-injection pattern was found only on `IGNITION_ROOT_KEY_PASSWORD` in the 8.3.6 docs; other secret injection paths through the file-config layer (specifically the JWE-stored `internal-secret-provider` entries) make `<VAR>_FILE` less necessary in practice. |
| `secret-reference` | yes | yes (with caveat) | A `config.json` referencing a secret looks like `{"<field>": {"data": {"providerName": "<provider>", "secretName": "<name>"}, "type": "Referenced"}}`. The reference itself is file-seedable. The secret value lives in the referenced provider's own file-seeded `config.json` as a JWE blob. **Caveat:** Phase 1 did not test writing a plaintext value into the secret-provider config to confirm whether 8.3.6 accepts plaintext + auto-encrypts on first read, so projects need either a known JWE value (lifted from a working seed) or to pre-encrypt their own values; this is a follow-up worth doing for Phase 2 ergonomics. |
| `module-install` | partial | n/a | See **Module install verdict** above. Placing `.modl` files under `/usr/local/bin/ignition/user-lib/modules/` (via any of options a/b/c) plus setting `GATEWAY_MODULES_ENABLED` to the module's fully-qualified identifier enables the module. Activation requires per-module certificate trust + license acceptance through the commissioning UI; no env-var bypass observed in 8.3.6. |

## Methodology

This investigation runs on `inductiveautomation/ignition:8.3.6` and walks three kinds of evidence, in order of how load-bearing they are for the matrix:

1. **Inspection of the reference template.** The `inductive-automation/template-ignition-project` repo is a validated, file-seeded Ignition project. The Phase 1 baseline at `scripts/seeding-poc/baseline/` boots a fresh 8.3.6 gateway with that template's `services/ignition/config/resources/` tree and `services/ignition/projects/` tree copied into the gateway data volume (a copy + `chown -R 2003:2003` is required because a `:ro` bind-mount of the resource trees faults when the gateway tries to create its `.resources/` cache directories). The `capture_baseline.py` Playwright harness then logs in and screenshots each subsystem config page, producing the evidence the matrix rows above point at.
2. **One live experiment for `q-module-install`.** Option (a) (runtime volume-mount of `MQTT-Engine-signed-5.0.2.modl`) was tested with `GATEWAY_MODULES_ENABLED` unset, set to the module identifier `com.cirruslink.mqtt.engine.gateway`, and compared against the docs description of the variable. The identifier form is what the variable accepts; the unset case leaves the module quiescent in `user-lib/modules/`. The certificate-trust + license-accept commissioning step is a new constraint not visible from the docs alone. Options (b) and (c) place the `.modl` at the same in-container path, so they behave identically for the load step and are documented as such rather than re-tested individually.
3. **Two investigative findings recorded but not separately experimented.** The `lastModificationSignature` validation question and the plaintext-secret-acceptance question are noted in the relevant rows as "needs follow-up" because the template's signed-and-encrypted artifacts decoded successfully on a fresh gateway (proving the no-validation / shared-key path works), which is sufficient to mark the relevant rows `yes` without a separate destructive experiment. Either question is straightforward to test in a future iteration if a downstream phase needs to write its own signatures or its own plaintext-then-encrypted secrets.

The Playwright assertion script at `scripts/seeding-poc/boot.py` automates Phase 1's headline claim end-to-end: it boots the baseline, logs in, navigates to Database Connections, and exits non-zero if the template's `db` connection does not appear with status VALID. That script is the regression gate this phase's downstream phases run against.
